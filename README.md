# mtls_ob_test — агент мониторинга OceanBase (прототип)

Лёгкий агент, который ставится на хост с OceanBase и отдаёт метрики наружу
по HTTP, а к самой базе ходит по mTLS. Идея — сервисы мониторинга не лезут в БД
напрямую: на хосте стоит агент, который выполняет только разрешённые запросы.

> Это **прототип / spike**. Часть защитных слоёв (аутентификация заливающего,
> ограничение по IP) ещё не реализована — см. «Ограничения».

---

## Модель безопасности (коротко)

- **На диске нет секретов.** Клиентский сертификат и ключ к базе живут только
  в оперативной памяти. На диске рядом с бинарём — лишь публичный `ca.pem`.
- **Сертификат заливается в рантайме.** Агент стартует «пустым» (фаза `NoCert`)
  и ждёт, пока ему передадут сертификат через управляющий эндпоинт.
- **Перезагрузка = чистый лист.** Рестарт стирает серт из памяти — украсть с
  машины нечего, агент снова ждёт заливки.
- **Доступ к базе — по сертификату (mTLS)** под отдельным пользователем БД с
  правами только на чтение.
- **Чистый Rust.** TLS на `rustls`/`ring`, без системного OpenSSL — один
  самодостаточный бинарь (зависит только от glibc), переносим на любой RHEL 9 /
  Rocky / Alma / Oracle 9.

---

## Архитектура

Два независимых сервера в одном процессе:

| Сервер | Протокол | Порт (по умолч.) | Назначение |
|--------|----------|------------------|------------|
| **data** | HTTP  | `8080` | `GET /version`, `GET /health` |
| **mgmt** | HTTPS | `9443` | `POST /cert`, `GET /cert/validity` |

mgmt-сервер слушает на **встроенном самоподписанном** сертификате (зашит в бинарь
через `include_str!`, файлы `embedded/mgmt-*.pem`). Это идентичность управляющего
канала, не секрет.

Пока сертификат к OB не залит — `/version` и `/cert/validity` честно отвечают
`503 {"error":"nocert"}`, `/health` отвечает `ok`.

## Запуск на новом сервере

Предполагается, что на хосте доступен OceanBase и есть твой `ca.pem`.

### Шаг 1. Выпустить клиентскую пару из своего CA

Нужны три файла (как выпускать — вне рамок этого README):

- `ca.pem` — корневой CA (кладётся рядом с бинарём);
- `client-cert.pem` + `client-key.pem` — клиентская пара агента, подписанная этим CA.

### Шаг 2. Создать пользователя в OceanBase под этот серт

OceanBase в `REQUIRE SUBJECT` ждёт **весь subject** сертификата в слэш-формате.
Достаём его из выпущенного серта:

```bash
openssl x509 -in client-cert.pem -noout -subject -nameopt compat
# subject=/C=SU/O=lab/OU=Database/CN=obcluster200
```

Берём строку после `subject=` и заводим пользователя (имя пользователя —
произвольное, оно же пойдёт в `OB_USER`; права — только чтение):

```sql
CREATE USER 'test_mtls'@'%'
  REQUIRE SUBJECT '/C=SU/O=lab/OU=Database/CN=obcluster200';
GRANT SELECT ON oceanbase.* TO 'test_mtls'@'%';
```

> `@'%'` пускает с любого IP — удобно для теста. Для прода сузь до конкретного
> хоста, например `'test_mtls'@'192.168.55.202'`.

### Шаг 3. Запустить агент

`ca.pem` лежит рядом с бинарём; клиентский серт на старте **не нужен** — зальём
позже. Настройки — через переменные окружения:

```bash
OB_HOST=192.168.55.202 \
OB_PORT=2881 \
OB_USER=test_mtls \
OB_CA=./ca.pem \
DATA_ADDR=0.0.0.0:8080 \
MGMT_ADDR=0.0.0.0:9443 \
./mtls_ob_test
```

| Переменная | По умолчанию | Описание |
|------------|--------------|----------|
| `OB_HOST`   | `192.168.55.202` | адрес OceanBase (должен быть в SAN серверного серта OB) |
| `OB_PORT`   | `2881` | порт OceanBase |
| `OB_USER`   | `nm_test` | пользователь БД (см. Шаг 2) |
| `OB_CA`     | `~/certpas/ca.pem` | корневой CA |
| `DATA_ADDR` | `0.0.0.0:8080` | адрес data-сервера (HTTP) |
| `MGMT_ADDR` | `0.0.0.0:9443` | адрес mgmt-сервера (HTTPS) |

Открой порты в firewall, если ходишь с другой машины:

```bash
firewall-cmd --permanent --add-port=8080/tcp
firewall-cmd --permanent --add-port=9443/tcp
firewall-cmd --reload
```

## Использование

После старта агент в фазе `NoCert`. Заливаем клиентскую пару и работаем.

### 1. Проверить, что агент жив, но серта ещё нет

```bash
curl -s  http://<host>:8080/version           # {"error":"nocert"}   (503)
curl -sk https://<host>:9443/cert/validity     # {"error":"nocert"}   (503)
curl -s  http://<host>:8080/health             # ok
```

### 2. Залить клиентский серт+ключ (два файла, обычный curl)

```bash
curl -sk https://<host>:9443/cert \
  -F cert=@client-cert.pem \
  -F key=@client-key.pem
```

Ответ при успехе:

```json
{
  "ok": true,
  "cn": "obcluster200",
  "not_before": "Jun 28 16:25:56 2026 +00:00",
  "not_after":  "Jun 28 16:35:56 2028 +00:00",
  "note": "серт принят и проверен подключением к OceanBase"
}
```

> Агент принимает серт **только** если он реально аутентифицировался в OceanBase
> (внутренний self-check запросом). Кривой/чужой серт будет отклонён, текущее
> состояние не меняется.

### 3. Агент ожил — метрики доступны

```bash
curl -s  http://<host>:8080/version            # {"version": "5.7.25-OceanBase_CE_CL-v4.4.2.0"}
curl -sk https://<host>:9443/cert/validity      # CN + срок текущего серта
```

### Горячая замена (ротация)

Повторный `POST /cert` с новой парой меняет серт на лету — `GET /cert/validity`
покажет новый срок, живые соединения не рвутся.

### Про `-k` и проверку mgmt-серта

`-k` отключает проверку самоподписанного mgmt-серта. «Правильно» — пинить его:

```bash
# отпечаток встроенного серта
openssl x509 -in embedded/mgmt-cert.pem -noout -fingerprint -sha256

# проверять mgmt по этому серту вместо -k
curl --cacert embedded/mgmt-cert.pem https://<host>:9443/cert/validity
```

## Ограничения (честно)

- **Аутентификации заливающего пока нет.** mgmt-эндпоинт защищён только серверным
  TLS — кто имеет доступ к порту, может залить серт. Следующий шаг — mTLS на mgmt
  (проверка client-cert центра по CA + сверка CN) и ограничение по IP управляющих
  серверов (firewall / app-level allowlist).
- **Защита серта в рантайме — это права ОС.** Серт живёт в памяти; кто может
  присоединить отладчик к процессу (тот же uid или root), тот его прочитает.
  Рекомендуется запускать под отдельным непривилегированным сервисным
  пользователем (`systemd`, `User=...`), закрыв ptrace для чужих.
- **Только Linux.** Бинарь собирается под Linux (glibc); musl-статика — опционально
  через контейнерную сборку.

## Структура

```
src/
  main.rs         — бутстрап, поднимает оба сервера
  config.rs       — конфиг из переменных окружения
  state.rs        — разделяемое состояние + горячая замена серта
  credentials.rs  — Identity (cert/key/ca в памяти)
  db.rs           — пул к OceanBase поверх mTLS, запросы
  api.rs          — data-сервер: /version, /health
  mgmt.rs         — mgmt-сервер: POST /cert, GET /cert/validity
  certinfo.rs     — разбор сертификата (CN, срок действия)
  error.rs        — единый тип ошибки API
embedded/
  mgmt-cert.pem   — встроенный серт mgmt-сервера
  mgmt-key.pem    — встроенный ключ mgmt-сервера
```
